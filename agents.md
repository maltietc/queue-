# Queue (OmniPost) - Context for AI Agents

**Внимание Агенту (AI):** Этот файл содержит весь необходимый контекст для понимания архитектуры, стека и текущего состояния проекта `Queue` (ранее `tg-scheduler`). Обязательно прочитай его перед началом работы, чтобы не нарушить существующие паттерны.

## 1. О проекте
**Queue** — это легковесный планировщик контента (SMM-комбайн) для мульти-канальной публикации. 
На текущий момент реализована полная поддержка Telegram (каналы и чаты), но архитектура базы данных специально спроектирована так, чтобы в будущем легко добавлять Instagram, YouTube, LinkedIn и другие платформы.

## 2. Технический Стек
- **Фреймворк:** Next.js 15 (App Router).
- **База Данных:** SQLite (через Prisma ORM). Файл базы: `prisma/dev.db`.
- **Стилизация:** Tailwind CSS.
- **Редактор текста:** Tiptap (Rich Text Editor). Настроен на выдачу чистого HTML, который затем парсится в специфичные форматы (например, MarkdownV2 для Telegram).
- **Фоновые задачи (Cron):** Реализованы через `node-cron` внутри `src/instrumentation.ts` (Next.js Instrumentation Hook).
- **Развертывание:** Готово к переносу на сервер через Docker (`Dockerfile` и `docker-compose.yml`). В `next.config.ts` включен `output: "standalone"`.

## 3. Архитектура Базы Данных (Prisma)
Система работает по принципу "Один пост — много публикаций".
Ключевые модели:
1. **Post:** Содержит сам контент поста (`content` в формате HTML), общую дату публикации (`publishAt`) и общий статус (`DRAFT`, `SCHEDULED`, `PUBLISHED`).
2. **Channel:** Интеграция с конкретной соцсетью. Имеет `name`, `platform` (сейчас только `TELEGRAM`), `platformId` (например, ID чата в ТГ) и флаг `isActive`.
3. **PostPublication:** Связующая таблица. Хранит статус публикации *конкретного* поста в *конкретном* канале. Статусы: `PENDING`, `SCHEDULED`, `SUCCESS`, `ERROR`. Хранит тексты ошибок (`errorMsg`) и ID сообщения в соцсети (`externalId`).

## 4. Ключевые механизмы (Важно для отладки)
- **Отложенный постинг (Cron):** 
  Файл `src/instrumentation.ts` запускается один раз при старте сервера Node.js. Он ежеминутно ищет `PostPublication` со статусами `SCHEDULED` или `PENDING`, где время публикации уже наступило.
  > ⚠️ **Критический нюанс Next.js:** При изменении кода в `instrumentation.ts` в режиме разработки, сервер разработки кэширует его. Чтобы применить изменения в планировщике, ОБЯЗАТЕЛЬНО нужно выполнить `rmdir /s /q .next` (или удалить папку `.next` вручную) перед перезапуском `pnpm run dev`.
- **Парсинг в Telegram:**
  Файл `src/lib/telegram/parser.ts` конвертирует HTML от Tiptap в Telegram MarkdownV2. Все спецсимволы (точка, дефис, скобки) экранируются по строгим правилам Telegram API. Если возникает ошибка `Bad Request: can't parse entities`, проблема на 99% в экранировании MarkdownV2.
- **Картинки:** Загружаются локально в `public/uploads`.

## 5. Работа с Git и Сохранение изменений
Проект привязан к публичному репозиторию на GitHub (`maltietc/queue-`).
Поскольку репозиторий публичный, **очень важно не скомпрометировать данные пользователя**.

- Файл базы данных (`prisma/dev.db`), загруженные файлы (`public/uploads/*`) и переменные окружения (`.env`) **ДОБАВЛЕНЫ В `.gitignore`**. Их нельзя пушить.
- **Инструкция по сохранению (для Агента):**
  Когда пользователь просит "закомитить" или "сохранить" изменения, используй стандартные команды:
  ```bash
  git add .
  git commit -m "Краткое описание изменений"
  git push origin main
  ```
  Не удаляй `.gitignore` и всегда проверяй `git status` перед коммитом крупных изменений, чтобы убедиться, что `.env` или `.db` файлы не отслеживаются.
